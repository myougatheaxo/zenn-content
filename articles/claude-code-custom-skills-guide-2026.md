## Claude Codeの「スキル」とは

Claude Codeには**カスタムスキル機能**があります。`.claude/skills/` ディレクトリにMDファイルを置くことで、`/コマンド名` で呼び出せる専用の命令セットを追加できます。

毎回同じ長いプロンプトを書く手間がなくなり、チーム全員が同じレビュー基準を使えます。

```
.claude/
└── skills/
    ├── code-review/
    │   └── SKILL.md    ← このファイルが /code-review のロジック
    └── secret-scanner/
        └── SKILL.md    ← /secret-scanner のロジック
```

---

## 実際にコードレビュースキルを作ってみる

### SKILL.mdの基本構造

```markdown
# スキル名 — 一行説明

## 概要
何をするスキルか

## トリガー条件
いつ呼ばれるか

## 実行手順
1. ステップ1
2. ステップ2

## 出力フォーマット
どんな形式で出力するか
```

### 実装例：/code-review スキル

5軸（設計・可読性・パフォーマンス・セキュリティ・テスト容易性）でコードをレビューするスキルを実装します。

**`.claude/skills/code-review/SKILL.md`** を作成:

```markdown
# code-review — 5軸コードレビュースキル

## 概要
コードを設計・可読性・パフォーマンス・セキュリティ・テスト容易性の5軸でレビューする。
重大度別に問題を分類し、修正コードサンプル付きで出力する。

## トリガー条件
- ユーザーが「コードレビュー」「レビューして」と言ったとき
- `/code-review` を明示的に呼び出したとき

## 実行手順

### 1. スコープ確認
指定ファイル/ディレクトリ/変更差分を対象とする

### 2. 5軸チェック
#### 軸1: 設計（SRP・OCP・依存性注入）
#### 軸2: 可読性（変数名・関数長・ネスト深さ）
#### 軸3: パフォーマンス（N+1・ループ内重複計算）
#### 軸4: セキュリティ（インジェクション・ハードコード認証情報）
#### 軸5: テスト容易性（グローバル状態・外部依存の分離）

### 3. レポート出力
重大度 CRITICAL/HIGH/MEDIUM/LOW で分類し、
修正コードサンプル付きで出力する。

## 注意事項
- コードを実際にRead/Grepで読んで確認する（推測禁止）
- テストファイル・コメント内のパターンマッチは除外する
```

### 使ってみる

```bash
# プロジェクトのAPIディレクトリをレビュー
/code-review src/api/

# 直前のコミット差分をレビュー
/code-review --diff
```

**出力例**:
```
## コードレビューレポート
対象: src/api/users.py

### 総合スコア: C
| 軸 | 評価 | 問題数 |
|----|------|--------|
| 設計 | B | 1件 |
| 可読性 | C | 3件 |
| パフォーマンス | D | 1件 |
| セキュリティ | A | 0件 |
| テスト容易性 | B | 1件 |

---

### [HIGH] 軸3-パフォーマンス: N+1クエリ
**ファイル**: `src/api/users.py:45`
**コード**:
for user in users:
    posts = db.query(f"SELECT * FROM posts WHERE user_id={user.id}")

**修正**:
users_with_posts = db.query(
    "SELECT u.*, p.* FROM users u LEFT JOIN posts p ON u.id=p.user_id"
)
```

## もう少し複雑なスキル：/secret-scanner

APIキーの漏洩を検出するスキルです。誤検知フィルタリング付き。

```markdown
# secret-scanner — シークレット漏洩検出

## 実行手順

### 1. パターンスキャン
以下の形式を Grep で検索する:
- AWS: `AKIA[0-9A-Z]{16}`
- GitHub: `ghp_[a-zA-Z0-9]{36}`
- Anthropic: `sk-ant-api\d{2}-[a-zA-Z0-9_-]{86}`
- Stripe: `sk_(live|test)_[a-zA-Z0-9]{24}`
- JWT: `eyJ[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+`

### 2. 誤検知フィルタリング
以下は除外する:
- テストファイル（*.test.*, spec*, fixture*）
- コメント行（#, //）
- プレースホルダー（YOUR_KEY_HERE, REPLACE_ME, example, dummy）

### 3. エントロピー分析
残った候補のエントロピーを計算。低エントロピー（< 3.5 bit/char）は除外

### 4. 出力
発見した箇所を重大度 CRITICAL で報告し、.gitignore 追加と環境変数移行を提案
```

## スキル設計のコツ

### 1. 「コードを読む」を明示する

```markdown
## 重要な注意事項
- コードを実際にRead/Grepで読んで確認する
- 推測でのレポート禁止
```

これを書かないと、Claude がファイルを読まずに「推測」でレポートを作ることがあります。

### 2. 出力フォーマットを具体的に定義する

テーブル、コードブロック、重大度ラベルなど、出力形式を詳細に指定すると安定します。

### 3. トリガー条件を複数書く

```markdown
## トリガー条件
- ユーザーが「セキュリティ」「脆弱性」「audit」と言ったとき
- `/security-audit` を明示的に呼び出したとき
- PRレビューでセキュリティ観点を求められたとき
```

自然言語でも呼び出せるようにすると、コマンドを忘れても機能します。

### 4. エラーケースを書く

```markdown
## エラーケース
- 対象ファイルが存在しない → 「ファイルが見つかりません。パスを確認してください」
- 言語が未対応 → 「対応言語: JS/TS/Python/Kotlin。他言語は部分的にチェック」
```

---

## まとめ

Claude Codeのカスタムスキルは：

1. `.claude/skills/<スキル名>/SKILL.md` を作るだけ
2. `実行手順` に具体的なステップを書く
3. `出力フォーマット` でレポート形式を定義する
4. 「コードを読む」「誤検知を除外する」などの注意事項を書く

スキルは再利用可能なので、チームで共有したり、プロジェクトをまたいで使えます。
