## はじめに

ドキュメントは書くのが面倒で、常に陳腐化する。Claude Codeを使うと、コードからドキュメントを生成できる。

ただし「ドキュメントを書いて」と丸投げするだけでは質が低い。制約を与えることで有用なアウトプットが得られる。

---

## APIドキュメントの生成

ルートハンドラーやコントローラーのファイルを渡す。

```
以下のAPIエンドポイントのOpenAPIドキュメントを生成してください。

含めるもの：
- エンドポイントパスとHTTPメソッド
- リクエストボディスキーマ（バリデーションコードから推測）
- レスポンススキーマ（返却型から推測）
- エラーレスポンス（400, 401, 404, 500）

[ルーターファイルをここに貼る]
```

TypeScript + Expressの場合：

```
このルーターの各ハンドラー関数にJSDocコメントを追加してください。
@paramはrequestの型、@returnsはレスポンスの型、@throwsはエラーケースを記載。

[ルーターファイルをここに貼る]
```

## README生成

```
このプロジェクトのREADME.mdを生成してください。

含めるもの：
- 一文の説明
- 必要条件
- インストール手順（package.jsonの実際のコマンドを使う）
- 環境変数の設定（.env.exampleから）
- ローカル実行方法
- テスト実行方法
- プロジェクト構成（トップレベルのディレクトリのみ）

コンテキスト：
[CLAUDE.mdまたはpackage.jsonをここに貼る]
```

ゼロから書くのではなく、コンテキストを埋める作業になる。

## コメントの生成

複雑なロジックへのコメント追加：

```
この関数にコメントを追加してください：
1. 関数の概要（1行）
2. アルゴリズムの説明（重要な手順のインラインコメント）
3. 自明でない設計判断の説明

自明な行にはコメントしないでください。
ロジックが明確でない箇所にのみコメントを追加してください。

[関数をここに貼る]
```

**「自明な行にはコメントしないでください」が重要。**
これがないと、こんなコメントが生成される：

```javascript
// iを1増やす
i++;
```

[git log --onelineの出力をここに貼る]

## アーキテクチャ決定記録（ADR）

なぜその技術を選んだかを記録するのがADR。

```
PostgreSQLの代わりにMongoDBを採用することを決定しました。
この決定のADR（アーキテクチャ決定記録）を生成してください。

含めるもの：
- 背景（解決する問題）
- 検討した選択肢
- 決定
- 結果（良い点と悪い点）
```

`docs/adr/` に保存し、CLAUDE.mdに参照を書いておくと、過去の決定をClaude Codeが考慮してくれる。

---

## CLAUDE.mdにドキュメントルールを書く

```markdown
## ドキュメントルール

### コメント追加基準
- アルゴリズムの説明（このアプローチを選んだ理由）
- 自明でないビジネスロジック
- 回避策（何を回避しているかの説明）
- 自明なコードにはコメント不要

### JSDoc/Docstring
- パブリック関数・エクスポートされた型には必須
- 複雑なプライベート関数にも推奨
- 単純なプライベートヘルパーには不要

### README
- APIに変更が入ったら `docs/openapi.yaml` を更新
- アーキテクチャの決定は `docs/adr/` に記録
```

---

## まとめ

Claude Codeでのドキュメント生成：

1. **APIドキュメント**: ルーターファイルを渡してOpenAPI/JSDocを生成
2. **README**: コンテキスト（CLAUDE.md等）を渡して生成
3. **コメント**: 「自明な行にはコメントしない」制約を付ける
4. **ADR**: 技術選択の理由を記録して将来のClaude Codeに伝える
