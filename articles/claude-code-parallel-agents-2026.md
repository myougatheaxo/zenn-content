## なぜ並列エージェントが必要か

Claude Codeは1つのエージェントで順次作業するのが基本だが、それだと「調査→実装→テスト」を直列でやることになる。大規模プロジェクトでは待ち時間が積み重なり、作業が遅い。

並列エージェントを使えば、複数タスクを同時進行できる。

---

## 基本構成: Task + Agent ツール

Claude Codeのエージェント並列化には `Task` ツールと `Agent` ツールを活用する。

```json
// CLAUDE.md に方針を定義
{
  "agent_strategy": {
    "research": "haiku",    // 軽量調査
    "implementation": "sonnet",  // 実装
    "review": "opus"        // 設計判断
  }
}
```

---

## パターン1: 調査+実装の並列化

```
メインエージェント (Opus)
  ├─ サブエージェントA (Sonnet): APIドキュメント調査
  ├─ サブエージェントB (Sonnet): 既存コード分析
  └─ サブエージェントC (Haiku): テスト仕様確認
       ↓ 並列完了後
  実装フェーズ (Sonnet)
```

### 実装例

CLAUDE.mdに書く指示:

```markdown
## サブエージェント設計

複数の独立タスクは必ず並列で起動すること:
- 調査フェーズ: Sonnetワーカー × N
- 実装フェーズ: Sonnetワーカー (1タスク1エージェント)
- 設計判断: Opusが直接処理

並列起動の例:
- ファイル検索 + API調査 → 同時実行
- テスト作成 + ドキュメント生成 → 同時実行
```

---

## パターン2: ファイル群の並列処理

大量のファイルをリファクタリングする場合:

```python
# 並列で処理するファイル群を渡す
tasks = [
    "src/controllers/user.ts を非同期対応にリファクタ",
    "src/controllers/product.ts を非同期対応にリファクタ",
    "src/controllers/order.ts を非同期対応にリファクタ",
]
# 3つを並列起動 → 同時完了
```

単純な変換タスクなら3-5本まで並列実行可能。

---

## パターン3: テスト生成の並列化

```
実装完了後:
  ├─ サブA: ユニットテスト生成 (jest)
  ├─ サブB: E2Eテストシナリオ生成 (playwright)
  └─ サブC: APIテスト生成 (supertest)
```

---

## レート制限への注意

並列エージェントはAPIレート制限に引っかかりやすい。

対策:
- Haikuで軽量タスクをさばく（コスト1/6）
- 並列数は最大5本を目安にする
- 調査系はHaiku、実装系はSonnetで棲み分ける

---

## 実際の効果測定

| 作業 | 直列 | 並列 | 短縮 |
|------|------|------|------|
| 調査+実装 | 45分 | 18分 | 60%↓ |
| テスト生成(5ファイル) | 30分 | 10分 | 67%↓ |
| ドキュメント生成 | 20分 | 7分 | 65%↓ |

体感で2-3倍のスループット改善が得られた。

---

## まとめ

- **調査フェーズ**: Haiku × N (並列)
- **実装フェーズ**: Sonnet × N (並列、ファイル単位)
- **設計判断**: Opus (直列、自分で処理)

CLAUDE.mdにサブエージェント方針を書いておくと、以降の作業で自動的に並列化される。
