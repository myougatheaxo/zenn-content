## 技術的負債をAIで解消する前に考えること

「Claude Codeに全部リファクタしてもらえばいい」と思うと失敗する。

技術的負債の解消は:
1. **何が問題か特定する** (負債の棚卸し)
2. **優先順位をつける** (ビジネスインパクト × 修正コスト)
3. **段階的に修正する** (一気にやると壊れる)

Claude Codeはこのプロセスを加速するツールであって、判断を代替するものではない。

---

## ステップ1: 負債の棚卸し (Claude Codeで自動化)

```bash
# 技術的負債の可視化
claude "以下の観点でコードを分析して負債リストを作成:
1. 循環的複雑度が高い関数 (>10)
2. テストカバレッジが0%のモジュール
3. 非推奨ライブラリの使用箇所
4. コメントアウトされたコード
5. TODO/FIXME/HACKコメント

src/以下を対象に、深刻度(高/中/低)と影響範囲を含めてMarkdownで出力"
```

出力例:
```markdown
## 技術的負債棚卸し (2026-03-11)

### 高優先度
- `src/payment/processor.ts`: 複雑度18、テストゼロ、本番影響大
- `src/auth/session.ts`: deprecated jwt-simple使用、セキュリティリスク

### 中優先度
- `src/utils/format.ts`: 重複ロジック6箇所、保守コスト高
...
```

## ステップ2: 優先順位マトリクス

```
         ビジネスインパクト
         高          低

     ├──────────┼──────────┤
     │  計画的  │  無視    │
高   │  対応    │  (今は)  │
     └──────────┴──────────┘
```

## ステップ3: 安全なリファクタリング手順

Claude Codeに一気にやらせると、動かなくなる可能性がある。

### 黄金律: テストがないコードは先にテストを書く

```bash
# まずテストを書かせる
claude "src/payment/processor.ts の現在の振る舞いを
E2Eテストとして記述。リファクタ前の動作保証が目的。
テストが通ってから実装を変更する"
```

### Strangler Figパターン

```bash
# 新しいモジュールを作って徐々に切り替え
claude "src/auth/session.ts を廃止するため:
1. src/auth/session-v2.ts を新規作成 (jose使用)
2. session.ts からのインポートを1ファイルずつ v2 に切り替え
3. 全テスト通過後に session.ts を削除
今回は手順1のみ実施"
```

## ステップ4: 自動化できる定型リファクタ

```bash
# console.log → 構造化ログへの一括置換
claude "src/以下の console.log を logger.info に置換。
logger は src/utils/logger.ts からインポート。
エラーハンドリング内の console.error は logger.error に"

# async/await への変換
claude "src/api/ 以下の Promise.then チェーンを
async/await に変換。エラーハンドリングは try/catch で"

# TypeScript strict mode対応
claude "tsconfig.json の strict: true を有効化。
発生する型エラーを解消。anyの使用は最小限に"
```

## 進捗トラッキング

```bash
# 週次で負債量を計測
claude "今週のリファクタ進捗をまとめて:
- 解消した負債数
- 残った負債数
- テストカバレッジの変化
- 複雑度の変化"
```

## よくある失敗パターン

| 失敗 | 回避策 |
|------|--------|
| テストなしでリファクタ | 先にテスト作成を必須化 |
| 一度に大量変更 | PR単位を小さく保つ |
| 「負債ゼロ」を目指す | ビジネスに影響する負債に絞る |
| レビューなしでマージ | AIのリファクタも必ずレビュー |

---

## まとめ

Claude Codeは「棚卸し → テスト作成 → 段階的修正」のサイクルを高速化する。一気にやろうとせず、テストが保証する範囲内で少しずつ進めるのが成功のコツ。
