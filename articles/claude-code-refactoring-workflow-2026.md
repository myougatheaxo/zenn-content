## はじめに

「リファクタリングしたい、でもこの大きなファイルをどこから手を付ければいいかわからない」

そういう場面でClaude Codeをどう使うか。ただ「リファクタリングして」と渡すだけでは、望ましくない変更が紛れ込むリスクがある。

---

## ステップ1: まず「診断」をさせる

修正より先に、現状の問題点を洗い出させる。

```
以下のファイルをリファクタリング候補の観点で分析してください。
具体的にどこが問題で、何が改善できるか教えてください（まだ修正はしないでください）：

[コードをここに貼る]
```

「まだ修正しないで」という一言が重要。先に診断結果を確認してから、どこを直すか選べる。

---

## ステップ2: スコープを絞る

大きなファイルを一気にリファクタリングしない。問題箇所を絞って指示する。

```
`UserService` クラスの中で、`processPayment` メソッドだけをリファクタリングしてください。
このメソッドは現在300行あり、複数の責任が混在しています。
変更箇所以外は触らないでください。
```

「変更箇所以外は触らないでください」はCLAUDE.mdにも書いておくと効果的。

---

## ステップ3: テストを先に書かせる

既存のテストがない場合、先にテストを書かせてからリファクタリングする。

```
以下のコードをリファクタリングする前に、現在の動作を保証するテストを書いてください：

[コード]
```

テストが通った状態でリファクタリングすれば、動作変更のリスクを防げる。

---

## ステップ4: 差分を確認させる

```
リファクタリング前後の差分を説明してください：
1. 何を変えたか
2. なぜその変更が改善になるか
3. 動作に影響する変更があるか
```

特に3番目の確認が重要。「動作に影響する変更があるか」を明示的に聞かないと、見落とされることがある。

---

## CLAUDE.mdにリファクタリングルールを書く

```markdown
## リファクタリングルール
- 明示された箇所以外は変更しない
- 関数を分割する場合は既存関数の動作を保持する
- 型定義を勝手に変更しない
- 外部から見えるAPIシグネチャを変更する場合は明示的に許可を求める
- リファクタリング後は既存テストが全て通ることを確認
```

## /refactor-suggestスキルの活用

カスタムスキルを使うと、定型的なリファクタリング提案を自動化できる。

```bash
/refactor-suggest src/services/payment.service.ts
```

出力例：
```
## リファクタリング提案

### 1. 関数の分割 (推奨度: 高)
`processPayment()` は3つの責任が混在しています：
- バリデーション → validatePaymentInput()
- 支払い処理 → executePayment()
- 通知送信 → sendPaymentNotification()

### 2. マジックナンバーの定数化 (推奨度: 中)
行42: `if (amount > 1000000)` → MAX_PAYMENT_AMOUNT = 1_000_000

### 3. エラーハンドリングの統一 (推奨度: 中)
try/catchブロックが3箇所でパターンが異なります
→ withErrorHandling() ラッパーで統一
```

## よくあるリファクタリング失敗パターン

| 失敗パターン | 回避方法 |
|---|---|
| 一気に大きな変更 | スコープを1関数に絞る |
| テストなしで変更 | 先にテストを書かせる |
| API変更を見落とす | 外部インターフェースの変更を明示的に確認 |
| 型定義が変わる | 変更前後の型を比較させる |

---

## まとめ

Claude Codeで安全なリファクタリング：

1. **診断先行**（修正より先に問題点を洗い出す）
2. **スコープを絞る**（一度に1関数・1クラス）
3. **テスト先行**（リファクタ前にテストを書く）
4. **差分確認**（動作変更がないか明示的に確認）
