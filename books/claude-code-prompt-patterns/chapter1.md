## パターン1: 「ゴール→制約→出力形式」の三段構成

**いつ使う？**
何か新しいコードを書いてもらうとき全般。これが全パターンの土台になるよ。

**プロンプト例:**

```
ユーザー登録APIを作って。

【ゴール】
- POST /api/users でユーザーを新規登録
- バリデーション付き

【制約】
- Express + TypeScript
- パスワードはbcryptでハッシュ化
- 既存のsrc/routes/配下に追加
- 既存のUserモデル(src/models/user.ts)を使うこと

【出力形式】
- ルートファイル、バリデーション、テストの3ファイルを作成
- 各ファイルにはインラインで簡潔なコメント
```

**ポイント:**
ゴール・制約・出力形式を明示すると、Claude Codeが「何を」「どう」「どんな形で」作ればいいか迷わない。制約がないと、自分の好みと違う技術スタックで作られちゃうことがあるんだよね。

**実行結果の例:**
3ファイルが正確に生成される。制約を書かない場合と比べて、手戻り修正が激減するよ。

## パターン2: 「既存コードを読んでから変更して」パターン

**いつ使う？**
既存コードベースに変更を加えるとき。新規ファイル追加じゃなくて、修正・拡張するとき。

**プロンプト例:**

```
src/services/payment.ts を読んで、以下の変更を加えて。

1. まず現在のコードを読んで構造を理解して
2. processPayment関数に、タイムアウト処理（30秒）を追加して
3. タイムアウト時はPaymentTimeoutErrorをthrowして
4. 既存のエラーハンドリングパターンに合わせて
5. 既存のテストファイルがあれば、そこにタイムアウトのテストも追加して
```

**ポイント:**
「まず読んで」を明示するのが大事。これがないと、Claude Codeがファイルの中身を確認せずに推測で書き換えることがあるんだよね。「既存のパターンに合わせて」も重要で、プロジェクト内で統一感のあるコードになるよ。

**実行結果の例:**
既存のtry-catchパターンに沿ったタイムアウト処理が追加され、テストも既存のdescribeブロック内に追加される。

## パターン3: 「テストを先に書いてから実装」パターン

**いつ使う？**
品質が重要な機能を実装するとき。TDD（テスト駆動開発）をClaude Codeにやらせるパターン。

**プロンプト例:**

```
カート機能の割引計算ロジックを実装したい。

手順:
1. まず以下の仕様に基づいてテストを書いて
   - 通常価格: そのまま合計
   - 10%割引クーポン: 合計から10%引き
   - 数量割引: 3個以上で5%引き
   - クーポン+数量割引: 両方適用（数量割引→クーポンの順）
   - 0円以下にはならない
2. テストを実行して全部失敗することを確認
3. テストが通る実装を書いて
4. テストを再実行して全部通ることを確認
```

**ポイント:**
テストを先に書かせることで、仕様が曖昧なまま実装されるのを防げるよ。Claude Codeはテスト→実装→再テストの流れをちゃんとやってくれる。仕様を箇条書きで具体的に書くのがコツ。

**実行結果の例:**
テストファイルが先に生成され、Red→Greenの流れで実装が完成。エッジケース（0円以下）もテストでカバーされた状態で納品される。
