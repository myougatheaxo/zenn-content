## パターン8: レビュー依頼 — 「セキュリティ観点でレビューして」

**いつ使う？**
自分が書いた（またはClaude Codeが生成した）コードの品質をチェックしたいとき。観点を絞ると精度が上がる。

**プロンプト例:**

```
src/api/ ディレクトリ内の全ルートハンドラをセキュリティ観点でレビューして。

チェック項目:
- SQLインジェクション（パラメータ化クエリを使っているか）
- XSS（出力エスケープされているか）
- 認証・認可チェック漏れ
- レートリミット設定
- 入力バリデーション（サイズ上限、型チェック）
- 機密情報のログ出力

出力形式:
- 深刻度（Critical / Warning / Info）
- ファイル名と行番号
- 問題の説明
- 修正案（コード付き）
```

**ポイント:**
レビュー観点を具体的に列挙する。「セキュリティレビューして」だけだと、一般的なことしか指摘してくれない。深刻度分けと修正案をセットで要求すると、指摘→修正がスムーズ。

**実行結果の例:**
「Critical: src/api/users.ts:45 — req.params.idがバリデーションなしでSQLクエリに結合されている。修正案: パラメータ化クエリに変更」のような具体的な指摘リストが出る。

## パターン9: リファクタリング — 「テスト壊さずに関数を分割して」

**いつ使う？**
巨大になった関数やファイルを、既存のテストを維持したまま構造改善したいとき。

**プロンプト例:**

```
src/services/order.ts の processOrder関数（現在200行以上）をリファクタリングして。

条件:
1. まず既存テスト(src/tests/order.test.ts)を実行して全部通ることを確認
2. processOrderを以下の責務で分割
   - validateOrder: 注文バリデーション
   - calculatePrice: 価格計算（割引・税込み）
   - createOrderRecord: DB保存
   - sendConfirmation: 確認メール送信
3. 分割後、元のprocessOrderはこれらを順番に呼ぶだけのオーケストレーターにする
4. 分割した各関数にも個別テストを追加
5. テストを再実行して全部通ることを確認
```

**ポイント:**
「テスト実行→リファクタ→テスト再実行」のサンドイッチにすることで、リファクタで壊れたらすぐ分かる。分割後の関数名と責務を指定することで、ぼくの意図と違う分割をされるのを防げるよ。

**実行結果の例:**
テスト12件全パス → 4関数に分割 → 新規テスト8件追加 → テスト20件全パス。processOrderは20行のオーケストレーターに。

## パターン10: ドキュメント生成 — 「JSDocを関数全部に追加して」

**いつ使う？**
コードは書いたけどドキュメントが後回しになってるとき。既存コードから自動生成させる。

**プロンプト例:**

```
src/utils/ ディレクトリ内の全.tsファイルにJSDocコメントを追加して。

ルール:
- exportされている関数・クラスのみ対象（内部関数は不要）
- @param と @returns は必須
- @throws がある場合は記載
- @example を1つ付ける（実際の使い方が分かるもの）
- 既にJSDocがある場合は上書きしない
- 説明文は日本語で簡潔に（1-2行）
```

**ポイント:**
「exportされているもののみ」で範囲を限定するのが大事。全関数にJSDocを付けると、内部実装の変更のたびにドキュメント修正が発生して逆に負債になるんだよね。「既存は上書きしない」も重要。

**実行結果の例:**
15関数にJSDocが追加。既存のJSDoc3件はそのまま維持。@exampleのコードは実際に動く形で記載される。

## パターン11: 型安全 — 「any型を全部適切な型に置き換えて」

**いつ使う？**
TypeScriptプロジェクトでany型が増えてきたとき。型安全性を一括で向上させる。

**プロンプト例:**

```
プロジェクト全体からany型を検索して、適切な型に置き換えて。

手順:
1. まずgrepで全anyの場所をリストアップ
2. 各anyについて、使われている文脈から適切な型を推測
3. 以下の優先度で型を決定:
   - 既存の型定義(types/やinterfaces/)があればそれを使う
   - API応答なら実際のレスポンス構造から型を作成
   - 本当に不明なものだけunknownにして、使用箇所にtype guardを追加
4. 変更後、tsc --noEmitでコンパイルエラーがないことを確認
5. any型の残数を報告
```

**ポイント:**
anyを全部unknownに変えるだけだと、使用箇所でキャストだらけになって逆に読みにくい。「既存の型定義を優先」「文脈から推測」を入れることで、実用的な型付けになるよ。

**実行結果の例:**
any 23箇所中、既存型で18箇所解決、新規型定義3件作成、unknown+type guard 2箇所。コンパイルエラー0。
