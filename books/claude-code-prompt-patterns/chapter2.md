## パターン4: バッチ処理 — 「このディレクトリの全ファイルを変換して」

**いつ使う？**
複数ファイルに同じ処理を一括で適用したいとき。手動でやると何十分もかかる繰り返し作業に。

**プロンプト例:**

```
data/csv/ ディレクトリ内の全.csvファイルに以下の処理をして。

1. 各CSVのヘッダー行を確認
2. 日付カラム(date列)のフォーマットを YYYY/MM/DD → YYYY-MM-DD に統一
3. 空行を除去
4. 処理済みファイルは data/csv/processed/ に出力（元ファイルは変更しない）
5. 処理結果のサマリー（ファイル名、行数、変換件数）を表示して
```

**ポイント:**
「元ファイルは変更しない」を明示するのが超重要。バッチ処理で元データを壊されたら泣くよね。出力先を分けて、サマリーも出してもらうと確認が楽。

**実行結果の例:**
Claude Codeがglob等でファイル一覧を取得し、1ファイルずつ処理。最後に「10ファイル処理完了、合計253行変換」のようなサマリーが出る。

## パターン5: 段階実行 — 「まず計画を立てて、承認したら実行して」

**いつ使う？**
大きな変更を加えるとき。一発で全部やらせると取り返しがつかないことがあるから、計画フェーズと実行フェーズを分けるパターン。

**プロンプト例:**

```
認証システムをJWTからセッションベースに移行したい。

まず計画だけ立てて。実行はしないで。

以下を計画に含めて:
- 影響を受けるファイルの一覧
- 各ファイルの変更概要
- 移行手順（順番が重要）
- リスクと対策
- ロールバック方法

計画を見てぼくがOKを出したら、その計画通りに実行して。
```

**ポイント:**
「実行はしないで」を明確に書く。Claude Codeは親切心で先に実行しちゃうことがあるから、計画フェーズでの出力を「計画のみ」に制限するのが大事。影響範囲が大きい変更ほどこのパターンが効く。

**実行結果の例:**
まず影響ファイル一覧と変更計画がMarkdownで出力される。内容を確認して「OK、この計画で実行して」と言えば、計画通りに順次実行。

## パターン6: 差分確認 — 「変更前後の差分を見せて」

**いつ使う？**
コード変更後に、何がどう変わったか確認したいとき。レビュー前のセルフチェックに最適。

**プロンプト例:**

```
さっきの変更について、以下の形式で差分サマリーを見せて。

1. 変更ファイル一覧（追加/変更/削除）
2. 各ファイルの変更概要（1-2行）
3. 重要な変更点のハイライト（ロジック変更やAPI変更）
4. 破壊的変更がある場合は警告
5. git diffの出力
```

**ポイント:**
`git diff`だけだと差分が大量で読みきれないことがある。概要サマリーとdiffの両方を出してもらうことで、全体像→詳細の順で確認できる。「破壊的変更の警告」を入れておくと、見落としを防げるよ。

**実行結果の例:**
「変更ファイル5件、うちAPI変更1件（破壊的変更: レスポンスのstatusフィールドがstringからenumに変更）」のようなサマリー付きで差分が出る。

## パターン7: 自動修正 — 「lint通るまで直して」

**いつ使う？**
コード生成後のlintエラーや型エラーを自動で解消させたいとき。手動でポチポチ直す時間を節約。

**プロンプト例:**

```
npm run lint を実行して、エラーが0になるまで修正して。

ルール:
- eslintのauto-fixで直るものは --fix で一括修正
- auto-fixで直らないものは手動で修正
- anyの使用は適切な型に置き換えて（不明なら unknown + type guard）
- disableコメントでの抑制は禁止
- 修正のたびにlintを再実行して、エラー0を確認するまで繰り返して
```

**ポイント:**
「disableコメント禁止」がないと、eslint-disableで全部握りつぶされることがある。「エラー0まで繰り返し」を明示するのも重要で、1回の修正で新たなエラーが出ることがあるから、ループさせるんだよね。

**実行結果の例:**
1回目: 15エラー → auto-fixで8件修正 → 手動修正7件 → 再実行: 2エラー → 修正 → 再実行: 0エラー。3ループで完了。
