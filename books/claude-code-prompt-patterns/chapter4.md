## パターン12: マルチファイル — 「APIとフロントとテストを同時に作って」

**いつ使う？**
1つの機能を実装するために、複数レイヤーのファイルを一度に作る必要があるとき。フルスタック開発で威力を発揮。

**プロンプト例:**

```
「ユーザープロフィール編集」機能を以下の構成で一括実装して。

バックエンド:
- PUT /api/users/:id/profile — ルートハンドラ
- src/services/profile.ts — ビジネスロジック
- src/validators/profile.ts — 入力バリデーション(zod)

フロントエンド:
- src/pages/ProfileEdit.tsx — 編集画面コンポーネント
- src/hooks/useProfile.ts — API呼び出しカスタムフック
- src/components/ProfileForm.tsx — フォームコンポーネント

テスト:
- tests/api/profile.test.ts — APIテスト
- tests/components/ProfileForm.test.tsx — コンポーネントテスト

共通条件:
- 既存のコードスタイルに合わせる（src/内の他のファイルを参考に）
- エラーハンドリングは既存パターンを踏襲
- フロント↔バックの型定義は shared/types/ に置いて共有
```

**ポイント:**
ファイル構成を明示的に指定する。Claude Codeに任せると、ファイル分割の粒度がぼくの想定と違うことがある。「既存のコードスタイルに合わせる」を入れることで、プロジェクト全体の一貫性が保たれるよ。

**実行結果の例:**
8ファイルが一括生成。共有型定義でフロントとバックの型が一致。テストも含めて1回のプロンプトで完了。

## パターン13: Git連携 — 「この変更をコミットしてPR説明文も書いて」

**いつ使う？**
開発が一段落したとき。コミット、ブランチ操作、PR作成までClaude Codeにやらせる。

**プロンプト例:**

```
現在の変更をコミットしてPRを作って。

手順:
1. git diffで変更内容を確認して表示
2. 変更内容に基づいてConventional Commitsの形式でコミットメッセージを作成
   例: feat(auth): add session-based authentication
3. コミットを実行
4. PR説明文を以下の形式で作成:
   ## 概要
   （1-2文で変更の目的）

   ## 変更内容
   - （箇条書き）

   ## テスト
   - （実行したテストと結果）

   ## 影響範囲
   - （他の機能への影響）
5. gh pr create で PRを作成
```

**ポイント:**
コミットメッセージの形式(Conventional Commits等)を指定しないと、毎回バラバラな形式になる。PR説明文のテンプレートを渡すことで、レビュワーが読みやすいPRが自動で作られるよ。

**実行結果の例:**
`feat(auth): add session-based authentication` でコミットされ、変更概要・テスト結果・影響範囲付きのPRが自動作成される。

## パターン14: デバッグ — 「このエラーの原因を特定して修正して」

**いつ使う？**
エラーが出ているが原因が分からないとき。エラー情報を渡して、調査→特定→修正まで一貫してやらせる。

**プロンプト例:**

```
以下のエラーを調査して修正して。

エラー内容:
TypeError: Cannot read properties of undefined (reading 'map')
  at UserList (src/components/UserList.tsx:23)

手順:
1. エラーが出ているファイルを読んで、該当行の周辺コードを確認
2. データの流れを上流に辿って、undefinedが渡される原因を特定
   - コンポーネントのprops
   - 親コンポーネントからのデータ渡し
   - API応答の構造
3. 根本原因を説明して
4. 修正を実施（応急処置ではなく根本修正）
5. 同じパターンのバグが他にないか確認
6. 修正後、関連するテストを実行
```

**ポイント:**
「応急処置ではなく根本修正」を明示する。これがないと、`data?.map`のようなオプショナルチェーンで握りつぶされることがある。「同じパターンのバグが他にないか」も入れると、類似バグの一網打尽ができるよ。

**実行結果の例:**
「原因: useEffectの初回レンダリング時にAPIレスポンスがまだ来ていないため、users=undefinedでmapが呼ばれている。根本修正: 初期値を空配列に設定 + ローディング状態の追加。同パターン2箇所追加修正」

## パターン15: 移行 — 「依存パッケージをv2からv3に移行して」

**いつ使う？**
ライブラリのメジャーバージョンアップで、破壊的変更への対応が必要なとき。

**プロンプト例:**

```
react-router を v5 から v6 に移行して。

手順:
1. package.jsonのバージョンを更新してインストール
2. v5→v6の破壊的変更一覧を把握（公式migration guideに基づく）
3. 影響を受けるファイルをリストアップ
4. 以下の優先度で段階的に移行:
   a. ルーティング定義（Switch → Routes, component → element）
   b. useHistory → useNavigate
   c. withRouter → useNavigate/useLocation/useParams
   d. 入れ子ルーティングの構文変更
5. 各段階でビルドが通ることを確認
6. 全体のテストを実行
7. 移行できなかった箇所があれば報告
```

**ポイント:**
移行は一気にやると壊れたときに原因特定が難しい。段階的に移行して各段階でビルド確認を入れることで、問題が起きてもすぐ分かるよ。「移行できなかった箇所の報告」も入れておくと、手動対応が必要な部分を見落とさない。

**実行結果の例:**
4段階で移行完了。ファイル12件変更。ビルド各段階でパス。テスト全通過。手動対応不要。
